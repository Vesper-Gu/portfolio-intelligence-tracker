# 需求文档

## 产品定位

Portfolio Intelligence Tracker 是一个个人使用的财经持仓知识库，用于追踪公开 KOL、文章、监管文件、截图和手动记录中的持仓信号，覆盖股票、加密资产、ETF、期权和其他资产类型。

本产品不是交易执行系统，也不提供投资建议。它的核心价值是保存可追溯的持仓观察记录，通过结构化数据、人工确认和后续语义检索，帮助用户复盘信息来源和持仓变化。

## 要解决的问题

有价值的持仓信号分散在推文、文章、SEC 文件、终端截图、券商 App 截图和个人备注中。个人投资者很难持续追踪 10 个以上信息源，也很难记住某个 KOL 何时加减仓、哪些人同时持有同一标的、某条历史观点来自哪里。

## 目标用户

- 第一阶段用户：个人研究者或投资者，自用为主。
- 后续潜在用户：小范围可信协作者组成的私有研究流程。
- 当前明确不做：公开多租户 SaaS。

## 核心场景

- 粘贴推文、文章片段或手动备注，保存为结构化持仓记录。
- 上传截图或后续接入截图解析，把解析结果先放入待确认队列。
- 解析结果需要人工确认后，才能进入正式持仓库；内部评分不作为第一版用户可见概念。
- 查看某个 KOL 的持仓详情、历史变化和来源上下文。
- 查看某个 ticker 被哪些 KOL 持有或操作。
- 跟踪质量指标，例如已验证比例、待确认数量、解析置信度。
- Phase 2 以后支持带来源引用的自然语言问答。

## Phase 1 范围

Phase 1 目标是先跑通核心数据闭环和最小 RAG 问答：

- KOL 和信息源管理。
- 支持 `stock`、`crypto`、`option`、`etf`、`other` 的手动持仓录入。
- 文本或截图解析候选结果进入 ingest queue。
- 人工确认流程：不确定记录确认后才成为正式 holdings。
- Supabase PostgreSQL schema：KOL、sources、holdings、snapshots、ingest items、alert rules、quality events。
- 单 KOL 和 ticker 级聚合 API。
- 数据质量汇总 API。
- 基于已确认 holdings、holding events、source ingest 和候选历史的最小 RAG 问答。
- 使用合成 seed data 和后端测试保障基础行为。

## 暂缓范围

- Embedding 和 pgvector 检索。
- embedding / pgvector 驱动的语义检索。
- 可配置 LLM 答案生成。
- LlamaIndex / FastAPI 服务。
- 财经新闻 RAG 联合检索。
- 多用户权限体系。
- 外部平台自动抓取或定时摄取。
- 生产级提醒通知投递。

## 用户流程

### 手动录入

1. 用户选择或创建 KOL / source。
2. 用户填写 ticker、资产类型、操作类型、可选仓位 / 排名 / 期权信息、原文、时间戳、备注和标签。
3. 系统规范化 ticker，校验字段范围，保存 holding，并标记为已验证。
4. 用户能在 KOL 详情页和 ticker 视图看到新记录。

### 解析确认

1. 用户粘贴文本或上传截图。
2. 系统保存原始输入和解析 payload，生成 ingest item。
3. 用户逐条查看解析结果，并决定加入资料库、修改、驳回或标记为需手动处理。
4. 被加入资料库的 item 生成正式 holdings，并保留来源上下文。

### 共识发现

1. 用户打开 dashboard 或 ticker matrix。
2. 系统按 ticker 聚合 holdings。
3. 用户看到 KOL 数量、平均仓位、加仓 / 减仓次数、最新时间和共识分数。

### RAG 查询

1. 用户输入自然语言问题，例如“@Investor_X 上次提到 TSLA 是什么时候？”
2. 系统基于用户已确认和已存入的数据检索相关证据，返回带引用的答案。
3. 用户可以打开每条引用对应的原始记录。

## 验收标准

- ticker 规范化会移除 `$`、转大写，并拒绝非法 ticker。
- 仓位百分比必须是 `0` 到 `100`，或为空。
- holdings API 返回统一 JSON envelope：成功为 `data`，失败为结构化 `error`。
- 共识聚合能基于已存 holdings 稳定排序 ticker。
- ingest items 在确认前必须保留原文 / 图片引用和解析 payload。
- 仓库中不得提交真实 API key、私有截图或生产用户数据。
- Phase 1 可以使用合成数据本地运行，不依赖 Supabase 凭证。
- 后端质量门禁通过：`npm --prefix backend test`。

## 产品指标

- Phase 1 数据目标：累计 100 条以上结构化 holdings。
- ticker 识别准确率目标：人工确认后大于 98%。
- 字段完整率目标：在来源质量允许时大于 80%。
- 人工确认流程目标：用户可以快速处理待复核记录，并明确区分已加入资料库和已驳回资料。
- RAG 指标：先关注是否能基于用户数据回答不同问题，embedding / LLM 质量指标推迟到后续阶段。

## 视觉方向

界面应像紧凑的研究工作台，而不是营销页。优先考虑高密度表格、清晰过滤器、来源可追溯和快速扫读。内部评分不默认展示给用户；图表用于辅助判断，但不能替代来源级检查。
